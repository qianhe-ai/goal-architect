# Goal Architect

把一句模糊需求整理成 agent 可以直接执行、可以验证、知道边界和预计耗时的目标。

它适合在 Codex、Claude Code、OpenClaw 等 agent 开工前使用：先把“帮我优化一下”“做一个系统”“睡前跑个长任务”这类粗略想法，整理成目标结果、执行范围、验证方式、预计耗时、约束、非目标、完成条件和暂停条件都清楚的任务说明。默认面向 Codex 输出 `/goal`，也可以输出通用 Agent 执行版。

<br>

## 安装

```bash
git clone https://github.com/qianhe-ai/goal-architect.git ~/.codex/skills/goal-architect
```


<br>

## 为什么要用它

给Agent派活，最普遍就2个问题：

1）目标不明确，做出来的不对

2）时间用不满，晚上8个小时，跑一会儿就结束了！

<br>

对于第一个问题，我们很容易说：

- “帮我把这个项目修一下。”
- “把这个 PPT 做得高级专业一点。”
- “做一个每天自动追踪 AI 资讯的系统。”
- “今晚睡觉规划一个尽可能长的任务，别几分钟就结束。”

但 agent 真正需要的是一份更像“任务合同”的目标：要交付什么、怎么证明完成、哪些文件或系统不能碰、失败后怎么迭代、什么时候要停下来问人，以及这件事大概会跑多久。

Goal Architect 的价值，就是把模糊想法先变成可执行目标，减少跑偏、早停、无意义扩展和高风险误操作。

对于第二个问题，如果你希望晚上睡觉前布置一个可以一口气尽量跑得长一点的任务，它也会先判断核心任务自然会跑多久，再基于上下文和记忆倒推可加入的扩展任务池，把多出来的时间用于真实有价值的验证、样例、二轮打磨、文档和风险复查。

它可以帮助你规划更清晰而长远的目标，充分利用好夜晚时间，大幅加速项目的推进速度。

<br>

## 它会怎么做

- 先判断用户真正关心的结果，而不是只复述表面动作。
- 把“高级、专业、完善、全面、稳定”等模糊词翻译成可检查证据。
- 基于低风险缺口做合理默认假设，让 agent 可以继续推进。
- 设计一轮更完整的执行范围：上下文读取、主体交付、验证、修正、文档和总结。
- 给出预计耗时，例如“约 30 分钟-2 小时”或“约 3-6 小时”。
- 如果用户给了期望时长，先保住核心交付，再倒推扩展任务池。
- 说明额外时间应该花在哪里，例如多源收集、样例、测试、视觉 QA、文档、发布检查和二轮修正。
- 明确边界、约束和非目标，避免 agent 顺手扩展到无关功能、生产系统、账号凭证或破坏性操作。
- 输出多行可读的 `/goal` 或通用 Agent 执行版，方便直接复制给 Codex、Claude Code、OpenClaw 或其他 agent。

最终结果通常包括：

- 规划结果
- 执行范围概览
- 预计耗时
- 可复制执行版 `/goal`
- 验证方式
- 约束、边界、非目标
- 迭代策略
- 完成条件
- 暂停条件
- 质量评估与风险

<br>

## 什么时候用

适合：

- 开工前把一句模糊需求整理成正式 agent 目标。
- 给 Codex 写 `/goal`，尤其是希望它一次性跑得更久、更完整时。
- 规划软件开发、项目修复、资料调研、文档生成、PPT/PDF 优化、自动化流程、知识库整理等多步骤任务。
- 睡前布置较长任务，希望知道预计耗时，并把额外时间用于有价值的验证和打磨。
- 已经知道希望任务大概跑 2 小时、4 小时或一整晚，希望反向规划出更完整但不跑偏的执行范围。
- 在公开发布、文件写入、自动化操作前，先明确边界和暂停条件。

不适合：

- 一句话翻译、改写、总结。
- 只需要跑一个简单命令的小事。
- 已经有非常明确执行步骤和验收标准的任务。
- 需要立即登录、支付、生产发布、删除数据或人工审批的任务。
- 用户真正想让 agent 直接执行，而不是先帮他完善目标的场景。

<br>

## 怎么用

在 Codex 里直接点名调用：

```text
用 $goal-architect 帮我把这个模糊需求完善成一个可以让 agent 一次性执行更久、更完整的目标：
我想做一个每天自动追踪全球 AI 资讯的系统。
```

也可以用于英文环境：

```text
Use $goal-architect to turn this rough request into a complete agent-ready goal:
Fix this repository so it can run locally, add minimal tests, and explain what changed.
```

如果你只想要可执行版本，可以直接说：

```text
用 $goal-architect，只输出可复制执行版。
```

如果你希望按时间倒推，可以这样说：

```text
用 $goal-architect，按 6-8 小时倒推规划这个目标：
我想让 agent 帮我把这个开源 skill 打磨到可以公开宣传和下载使用。
```

如果目标环境不是 Codex，可以要求输出通用版本：

```text
用 $goal-architect，把下面需求整理成 Claude Code / OpenClaw 可直接复制的通用 Agent 执行版，不使用 /goal：
我想让 agent 检查这个仓库、完善文档、补充验证，并给出最终交付说明。
```

<br>

## 示例

- `examples/ai-news-automation.md`：把“睡前做一个 AI 资讯系统，希望 agent 跑 6 小时”整理成带核心交付、扩展任务池、公开信源、样例日报、验证方式和预计耗时的目标。
- `examples/presentation-polish.md`：把“优化 PPT/PDF”整理成逐页排版、导出、渲染检查、二轮修正和交付说明目标。
- `examples/project-repair.md`：把“项目跑不起来”整理成复现、定位、最小修复、启动验证、预计耗时和暂停条件。

这些案例展示了从模糊输入到结构化 `/goal` 的转换方式，也可以作为你自己写目标时的参考模板。

<br>

## 内置规则

- 用户想要的结果优先，评分、边界和风险后置。
- 默认使用中文；只有用户要求或目标环境需要时才补英文版本。
- `/goal` 命令前缀保持不变，不改成中文命令。
- 面向 Claude Code、OpenClaw、OpenCode 或通用 agent 时，可以输出不依赖 `/goal` 的“目标 / 验证 / 预计耗时 / 约束 / 边界 / 非目标 / 迭代策略 / 完成条件 / 暂停条件”格式。
- 低风险信息缺口可以合理假设并继续推进。
- 高风险缺口必须暂停或明确写进暂停条件。
- 不执行生成出来的目标，除非用户明确要求继续执行。
- 不默认使用难读的长代码框承载完整目标，优先使用多行普通 Markdown。
- 预计耗时只给时间范围，不给“短任务/标准任务/深度任务”这类名称标签。
- 如果用户给了期望任务时长，会先判断自然耗时，再倒推核心交付和扩展任务池。
- 额外时间只能用于有价值的验证、样例、测试、文档、二轮修正和交付说明。
- 不为了“跑久一点”做无关搜索、无关重构或重复失败尝试。
- 每个多步骤目标都应包含验证、约束、边界、非目标、迭代策略、完成条件和暂停条件。

<br>

## 安全与边界

Goal Architect 只负责设计目标，不自动执行目标本身。

遇到以下风险时，生成的目标会要求暂停：

- 账号登录、验证码、凭证、token 或密钥。
- 付款、订阅、采购、计费或付费 API。
- 生产数据、生产部署、线上数据库或不可逆操作。
- 删除、覆盖、迁移重要文件或数据库。
- 法律、医疗、金融、合规等高风险判断。
- 版权不明的资料、图片、音视频或第三方内容。
- 公开发布、对外发送、社交媒体发布等外部动作。
- 用户意图、所有权、发布范围或预算不清。

处理方案：

- 低风险缺口：写明默认假设，继续生成可执行目标。
- 中风险缺口：写入约束、边界或非目标。
- 高风险缺口：写入暂停条件，要求用户确认后再执行。
- 已知敏感信息：不写进公开 README、示例或目标草稿。

## 环境与兼容

前置条件：

- 已安装支持 skills 的 Codex 环境。
- 本地存在 `~/.codex/skills` 目录，或允许创建该目录。
- 安装后可能需要重启 Codex 或刷新 skills 列表。

兼容说明：

- 主要面向 Codex skills 结构，默认生成 Codex `/goal`。
- Claude Code 支持包含 `SKILL.md` 的自定义技能目录，因此本项目的主要说明结构可以被 Claude Code 读取和改造。
- OpenClaw、OpenCode 和其他通用 agent 工作流的具体安装方式可能不同；遇到不支持 Codex `/goal` 命令的环境，可以要求本 skill 输出“通用 Agent 执行版”。
- 本 skill 不依赖外部 API、账号登录、付费服务或本地数据库。
