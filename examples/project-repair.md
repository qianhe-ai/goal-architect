# Example: Project Repair

## Input

```text
这个项目突然跑不起来了，你帮我修一下。
```

## Output Shape

规划结果：这次目标会让 agent 复现启动失败、定位根因、完成最小必要修复，并交付最终启动命令、日志证据、改动说明和剩余风险。

执行范围概览：

- 读取 README、脚本和项目约定
- 复现当前错误并保存日志
- 定位环境、配置、依赖或代码根因
- 做最小修复并重新验证
- 输出最终命令和风险说明

可复制执行版（多行普通文本）

/goal 修复当前项目启动失败问题，先复现错误并定位根因，再做最小必要改动，使项目能按现有文档或脚本在本地成功启动，并交付最终启动方式、验证日志、改动摘要和剩余风险。

验证：读取项目 README、package scripts、Makefile 或现有启动说明；运行当前推荐启动命令并保存错误日志；修复后重新运行同一启动命令，确认服务成功启动或关键检查通过，并记录最终命令、端口和剩余风险。

预计耗时：约 30 分钟-2 小时。根因跨依赖、环境和多文件代码时会接近上限；不通过无关重构来拉长时间，连续卡在凭证、私有依赖或系统级权限时应暂停。

约束：不做无关重构、不升级大版本依赖、不删除用户数据、不改生产配置、不改业务逻辑，除非根因证据证明必须这样做并先说明原因。

边界：只修改启动链路直接相关的配置、依赖声明、环境样例、脚本或代码文件；不触碰无关模块、部署配置、密钥文件、数据库数据和用户未授权文件。

非目标：不新增功能、不优化 UI、不重做架构、不处理启动失败以外的历史问题。

迭代策略：每次只验证一个根因假设；同一错误连续失败两次后必须换证据来源，例如查看完整日志、最小复现、依赖版本、环境变量或项目文档；最多做 3 轮聚焦修复后报告阻塞。

完成条件：项目能用记录的命令在本地成功启动，或最小相关检查通过；如果环境缺失导致无法完成，需明确列出缺失项、已验证内容和用户下一步操作。

暂停条件：需要真实生产凭证、删除数据、重装系统组件、修改远程服务器、升级关键框架大版本、访问私有仓库或做产品规则判断时暂停。
